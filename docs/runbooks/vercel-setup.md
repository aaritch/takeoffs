# Runbook: Vercel deployment setup

**Status: PAUSED.** Vercel is connected to the repo but there is **nothing deployable yet** —
`apps/web` is currently the backend domain layer only (no Next.js app, no routes/pages, no
`next` dependency). `pnpm run build` runs `turbo run build`, which only **typechecks**
(`tsc --noEmit`) and emits no output, so a Vercel build will fail at the "no output directory"
step (or publish an empty site).

**Do not enable Vercel deploys until the Next.js app shell exists (task P0-05).** To stop the
failing builds in the meantime, either disconnect the Git integration or set
Settings → Git → **Ignored Build Step** to exit 0 (e.g. `exit 0`).

When `apps/web` is a real Next.js app, configure the project as below.

## 1. Prerequisite

- `apps/web` is a Next.js (App Router) app: has `next` as a dependency, an `app/` directory
  with at least one route, and `next build` produces output. (Built in P0-05.)

## 2. Project settings (Vercel dashboard → Project → Settings)

- **Root Directory:** `apps/web`. Turn ON *"Include source files outside of the Root
  Directory in the Build Step"* so the workspace packages (`packages/*`) resolve.
- **Framework Preset:** Next.js (auto-detected once `next` is a dependency).
- **Build Command:** default `next build` works with Root Directory set; for Turborepo
  caching prefer `pnpm turbo run build --filter=@takeoff/web...` from the repo root.
- **Install Command:** leave default (Vercel installs the pnpm workspace from the root).
- **Node.js Version:** 22 (matches `.nvmrc`; `engines` allows ≥20).

## 3. pnpm / lockfile note

The first build warned: lockfile read as generated by pnpm 10 while `packageManager` pins
`pnpm@9.15.4`. Install still succeeded. To silence it, align `package.json#packageManager`
with the pnpm version actually used to write `pnpm-lock.yaml` (Vercel honors it via Corepack).

## 4. Environment variables (add as integrations are provisioned)

Add per environment (Production / Preview / Development). The full annotated list is in
[`.env.example`](../../.env.example). Prefer the **Vercel Marketplace integrations** (Neon,
Upstash, Blob) — they inject most of these automatically.

- **Database (Neon):** `DATABASE_URL` (admin/migrations role), `APP_DATABASE_URL`
  (**non-superuser** tenant role — required for RLS to actually enforce; mirror local
  `takeoff_app`).
- **Cache/queue (Upstash):** `REDIS_URL` (+ `UPSTASH_REDIS_REST_URL` / `_TOKEN`).
- **Object storage:** `BLOB_READ_WRITE_TOKEN` (Vercel Blob) or the `S3_*` set.
- **Auth (OIDC provider):** `AUTH_ISSUER_URL`, `AUTH_CLIENT_ID`, `AUTH_CLIENT_SECRET`,
  `AUTH_AUDIENCE`, `AUTH_JWKS_URL`, `AUTH_REDIRECT_URL`, `SESSION_SECRET`.
- **App:** `APP_BASE_URL`, `NEXT_PUBLIC_APP_BASE_URL`. (Phase 4: `STRIPE_*`, `EMAIL_*`.)

## 5. Migrations (NOT during build)

Run Drizzle migrations against the hosted DB as a **separate deploy step**, never inside
`next build`: `pnpm --filter @takeoff/web db:migrate` with the hosted `DATABASE_URL` (the
admin role). Seed global data once with `db:seed`.

## 6. Out of scope for Vercel

Workers (ingestion/raster/tile/export), the AI inference service, and the realtime WebSocket
gateway do **not** deploy to Vercel — they run on the Phase-2 compute host. Vercel hosts only
the Next.js app (UI + synchronous API).
